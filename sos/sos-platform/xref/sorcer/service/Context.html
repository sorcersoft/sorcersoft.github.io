35  /**
36   * Service context classes that implement this interface provide SORCER generic
37   * metacomputing data structures for storage, retrieval, and propagation of
38   * heterogeneous information across all SORCER service providers. Two generic
39   * implementations are provided: {@link ServiceContext} and
40   * {@link sorcer.core.context.PositionalContext}. Usually the former is used by
41   * service requestors and the latter with more functionality is used by service
42   * providers. The ServiceContextImpl class implements the ProviderContext
43   * interface that extends ServiceContext. An example of a specific service context
44   * is illustrated by {@link ArrayContext}.
45   * <p/>
46   * A service context is a tree-like structure with two types of nodes. Leaf
47   * nodes are called data (value) nodes and the remaining nodes are called
48   * context attribute nodes. Context attributes define a namespace for the data
49   * in a uniform way for use by all related services. A path of context
50   * attributes leading from the root of the context to any leaf node along with
51   * its data node is called a context element (path/data). Thus a path is a
52   * hierarchical attribute of data contained in the leaf node. A data node can
53   * contain any Java object; in particular a generic {@see ContextNode} is
54   * available.
55   * <p/>
56   * Context paths can be marked with attributes set by using the method
57   * {@link #setAttribute(String)}, which allow for efficient search of related
58   * data by service providers (service context clients). The search issue becomes
59   * critical when a namespace in the context may change or when data nodes
60   * contain remote references, for example a URL. It is usually assumed that
61   * provider-enforced data associations are more stable than user-oriented paths.
62   * Each direct service invocation {@link Service#service}
63   * requires data in the ServiceContext format.
64   * <p/>
65   * Service contexts are defined by this common interface with efficient
66   * service-oriented APIs. However, there are some similarities with XML
67   * terminology. The root element in XML contains all other XML elements.
68   * Similarly, in each service context, the root extends to all data nodes. A
69   * path with its data node (context element) is conceptually similar to an XML
70   * element. XML tags (markup) correspond to context attributes. A context data
71   * node can be considered the analog of data in XML, however a service context
72   * data node has rich OO semantics. Finally, data attributes in service contexts
73   * can be compared to element attributes in XML. However, data attributes have
74   * more meta-attribute meaning than XML attributes. While context attributes
75   * provide a name space for direct access to data nodes via
76   * {@link Context#getValue}, data attributes specify the data node for indirect
77   * efficient retrieval (search) by service providers. *
78   */
