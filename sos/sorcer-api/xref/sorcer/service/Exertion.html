32  /**
33   * An exertion is a federated service command (statement) communicated by a requestor -
34   * {@link Service#service}, to a service provider {@link Service}. It is a
35   * form of service-oriented message with an enclosed service
36   * {@link sorcer.service.Context} and a collection of service
37   * {@link sorcer.service.Signature}s. The service dataContext specifies service data
38   * to be processed by the service providers matching exertion's signatures. An
39   * exertion's signature can carry own implementation when its type is associated
40   * with a provided codebase.<br>
41   * Exertions are request interseptors in the form of command objects (in the
42   * Command programming pattern) that keep a service requestor completely
43   * separate from the actions that service providers carry out. Exertions are
44   * also completely separate from each other and do not know what other exertions
45   * do. However, they can pass parameters via contexts by mapping
46   * {@link Context#map} expected input dataContext values in terms of expected output
47   * values calculated by other exertion. The operation {@link Exertion#exert} of
48   * this interface provides for execution of desired actions enclosed in
49   * exertions, thus keeping the knowledge of what to do inside of the exertions,
50   * instead of having another parts of SO program to make these decisions. When
51   * an exertion is invoked then the exertion redirects control to a dynamically
52   * bound {@link Service} matching the exertion's signature of
53   * type Operate <code>PROCESS</code>. <br>
54   * The <code>Exertion</code> interface also provides for the Composite design
55   * pattern and defines a common elementary behavior for all exertions of
56   * {@link sorcer.service.Task} type and control flow exertions (for branching
57   * and looping). A composite exertion called {@link sorcer.service.Job} contains
58   * a collection of elementary, flow control, and other composite exertions with
59   * the same elementary behavior performed collectively by all its members. Thus
60   * {@link sorcer.service.Task}s and control flow exertions are analogous to
61   * programming statements and {@link sorcer.service.Job}s analogous to
62   * procedures in conventional procedural programming. <br>
63   * Control flow exertions allow for branching and looping operations by
64   * {@link Service}s executing exertions. A job combined from
65   * tasks and other jobs along with relevant control flow exertions is a
66   * service-oriented procedure that can federate its execution with multiple
67   * service providers bound dynamically in runtime as determined by
68   * {@link Signature}s of job component exertions.
69   * 
70   * @see sorcer.service.AsyncExertion
71   * @author Mike Sobolewski
72   */
